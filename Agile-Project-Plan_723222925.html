George Eastman Museum : Agile Project Plan

Created by Andrew Rosborough, last modified on Sep 24, 2018

Agile Process

Our Kanban based agile process entails collecting tasks and organizing work estimates to be performed in a workflow which enables a single developer to stay organized and focused on their deliverables.  As a developer becomes available to work on a new area of the project, they choose a story point or task from a queue to work on.  All of the tasks are organized on a Kanban board which shows columns for statuses of work such as work in progress, needs review and done.  The developer may only focus on a few issues through their completion before moving to a new task.  This is called limiting work in progress and allows for the time lost by re-focusing attention between different tasks to be mitigated.  The Agile process is unique because while utilizing kanban, developers choose sets of features to work on in a manageable timeframe such as one week.  These are delivered incrementally as initial prototypes and then more refined versions with each iteration of development.

Iterations and Sprints

A set of obtainable story points will be selected for development at the start of a sprint.  The sprint will consist of processing each subset of the backlog until all tasks are done.  An iteration is considered a revisitation of the features in an additional sprint where bugs and other tasks are identified to refactor an earlier sprint's functionality.

Release cycle

Initially, several sprints are performed and at a point deemed acceptable by the development team, an early beta is staged to a shared environment.  This beta version of the software is updated on a continuous basis as the product is developed.  Releases will be made on a regular basis to a Development environment or Standalone Feature Multi-dev environment and be merged in the manner of going from Feature to Dev to Test to Live.  Before the Launch Date, the furthest this workflow will be utilized will be to deploy to the Test environment for review of stable releases.

Organizing Epics, Users Stories, and Tasks and Bugs

When populating the kanban board, work is organized under Epics, User Stories, Tasks, and Bugs.  Generally, the work in the Kanban board is a set of living documents, which frequently change.  Epics describe broad features which include many Tasks, User Stories, and Bugs underneath them.  An appropriate Epic may be named after a section of the application, or a specific domain of functionality.  User Stories describe tasks the user wants to accomplish with the system.  This allows for full coverage of all business cases which are presented during discovery.  As action items come up that are not specifically covered by a full use case requiring a User Story, Tasks are created which specify the work being done.  Finally, as issues are moved through a review process Bugs are filed which get worked on to resolve the functionality of the initial issue.

Testing and Quality Assurance

Often Customers opt to use a service which takes screenshots of a set of pages on the website and compares them against former captures of the website to observe changes.  This is known as Visual Regression testing.  For sites which are heavy on the presentation layer of their content with minimal user customizations, it is recommended to rely on this automated process to catch glaring issues.

During a handoff phase, an issue collector may be enabled for the George Eastman Museum team to file QA issues for resolution by DropForge.

Maintenance and Ongoing Support

A help desk is utilized for customers with ongoing maintenance and support contracts.  Issues are typically responded to in under 24 hours and resolved within the terms of the contract.
