Content Types

Modified In Drupal 8 New Installation

Film Series will be replaced by a Content type to serve as a Series for the following sub-content types: Film Screening, Exhibition, Event.

The content types will be refactored to suit the needs of a robust calendaring solution which relies on core Drupal functionality and can be extended by adding additional sub-content types for inclusion by the Series content type.

Regarding recurring occurrences in a series; It is possible that if the date_recur contributed module is assisted with further development, that a reliable recurring date solution could be achieved.  The workflow would include creating a series containing one occurrence of an Event, Film Screening, or Exhibition which has a repeat rule stating which dates the occurrence repeats on, when the repeat rule should stop and which dates should be excluded.  Currently views support and interactive rule generation are in need of attention from the module maintainer.

Another approach to handling large repetitious series content would be to create a multistep form that generates Series content referencing generated sub-content while utilizing non-recurring core date fields. (e.g. For an event such as the Museum tours which repeat weekly).  A process involving a multistep form may include the following steps:

  • Indicate which days of the week should repeat
  • Exclude specific dates (e.g. for holidays or renovations)
  • Indicate a start date
  • Indicate an end date
  • Fill out a template of referenced sub-content from available Content Types (Event, Film Screening, Exhibition) up to 7 (One template per day of the week, sorted by weight.  Should validate that either 1 template exists for all days or the number of days and templates are equal)
  • Submit Form
  • Calculate Dates starting at start date working through to end date
  • Clone appropriate template (1 of up to 7) as a new node of similar type with default values from original node
  • Update date value on cloned content with the calculated value (Leave time untouched, only change Month or Day)
  • Save cloned content and append to reference field_occurrences on the parent Series.
  • Save the Series and redirect the user to view the newly generated completed Series

Forcing functions for Events, Film Screenings, and Exhibitions with Date fields

Date fields are used as the primary driver for navigation through the calendar.  In addition, presentational information can be shown with time and time_range fields, as they are configured to be visible only when 'All day' is not true, and time_range is not displayed when 'Collect an end date' is not true.  If 'Collect and end date' is true, then time is not visible and time_range becomes visible.  Additional work has to be done to make this compatible with inline entity field which allows for the inclusion of Events, Film Series, and Exhibition forms while editing a Series or adding a new Series.

Migration

Baseline

A fresh installation of Drupal 8, followed by enabling required modules for the project and at a minimum, any modules which were used in the legacy codebase which have Drupal versions or Drupal equivalents should be enabled, but not configured.  Default Migration scripts are to be generated using the Core Drupal UI on a fresh installation connected to the legacy database in addition to the Drupal 8 installation database.  These scripts will be exported as '.yml' and copied into a custom module tracked against the project's code base.

At this point, the baseline migration scripts will be used to make alterations either including or excluding fields and creating mappings from fields that do not have Drupal 8 equivalents to supported configuration for viable replacements as indicated in the Content Types section of this document.

Structure

Drupal configuration related to the admin/configuration and admin/structure areas of the CMS will be fine-tuned against the migration scripts as they are developed with the goal in mind being to enhance the usability of the fields which are being migrated.  At this stage, content types will be modified as indicated by Strikethrough text and Bold text in the Content Types section of this document.

Files

A legacy_files directory with the original file structure will be created outside of the docroot of the project.  Configuration for file fields will handle mapping the file table URIs to files included in the legacy file field.  These files will be copied into the new files directory as they are migrated individually.

Content

Testing will occur by running content migrations and rolling them back as errors are encountered until verification has been made that each content type is imported into the staging environment successfully.  After confirmation of all content types, the site will be staged to a hosting environment so that development may begin on the Theme and refactored series experiences.

Taxonomy

Taxonomy content will be migrated for vocabularies as appropriate.  The terms can be refactored after the migration by removing or creating additional taxonomy terms.

Field Collection

Search

A SOLR search index will be configured to handle offloading search queries from the database to a dedicated search index.  This will increase the performance of searching and offer advanced filtering capabilities.

As per requirements by many PaaS providers, a single Index will be created against the Node table and fields will be added which are indicated in the Content Types section of the report to the search index so that all field values and content types are in the index.  This will facilitate global search, and allow for relationships to be built in Views which are configured with the search index.  Additionally, the Title and any Text area and summary fields will be indexed as full-text search.  Taxonomy term reference fields will be indexed so as to use search facets which limit the query results by removing results from the query which do not contain the referenced taxonomy term in any of their field values.

Webforms

Webforms will be rebuilt in Drupal 8 but results and submissions will not be carried over.  There have been significant changes in Drupal's version of webform since Drupal 7.

Multi-step forms for Volunteer registration and other purposes will be built using web-form to suit the use-cases that have been provided.  The webform suite is highly configurable and can handle many business cases regarding form data collection and form presentation.
